# First admin

The bootstrap provisions the first cluster admin automatically. This page explains how, what variables control it, and how to recover if you end up locked out.

## How bootstrap binds cluster admin

The `apps` stack creates an `agyn_user` Terraform resource with `cluster_role = "admin"` and an `oidc_subject` taken from the `ADMIN_OIDC_SUBJECT` environment variable (default `admin@agyn.io`):

```hcl
resource "agyn_user" "admin" {
  oidc_subject = var.admin_oidc_subject
  name         = var.admin_oidc_subject
  cluster_role = "admin"
}
```

When you sign in via OIDC and the IdP returns an ID token whose `sub` claim matches `oidc_subject`, the Users service resolves you to this pre-created admin record. The cluster admin tuple was already written to OpenFGA by the `platform` stack against a deterministic identity ID, so the role takes effect immediately.

In addition, the `platform` stack mints a static **cluster admin API token** (a random password generated by Terraform) and stores it as the output `cluster_admin_api_token`. This token is used internally by the platform during bootstrap and is available if you need to perform admin operations via the Gateway API before signing in.

## Default vs. real OIDC

- **Default (mock OIDC).** `apply.sh` uses a public mock IdP and creates the admin user as `admin@agyn.io`. Sign in with that user on the mock IdP and you have cluster admin. This is intentional for fast evaluation; do not use the mock IdP for anything real.
- **Real OIDC.** Set `OIDC_ISSUER_URL`, `OIDC_CLIENT_ID`, `OIDC_CLIENT_SECRET`, and `ADMIN_OIDC_SUBJECT` (to your IdP-issued `sub` for the human or service that should be cluster admin) before running `apply.sh`.

```sh
export OIDC_ISSUER_URL=https://login.example.com/oidc
export OIDC_CLIENT_ID=...
export OIDC_CLIENT_SECRET=...
export ADMIN_OIDC_SUBJECT=auth0|abc123
./apply.sh
```

The admin user is then created against your real IdP subject.

## Verify

1. Open the Console (`https://console.agyn.dev:2496/` on bootstrap defaults).
2. Sign in via OIDC.
3. Open the context switcher (top-right). If you see **Cluster Administration**, you have the role.
4. Click into Cluster Administration → **Users** (`/users`). Your account should show `Cluster Admin = yes`.

If the role is missing, jump to [Recovery](#recovery).

## What the role grants

| Capability | Where it shows up |
|---|---|
| Manage platform users | Cluster Administration → Users |
| Grant or revoke cluster admin to other users | same view |
| Register cluster-scoped runners | Cluster Administration → Runners |
| Oversee all organizations | Context switcher lists every org |
| Inherit owner-level access on every org | All org-level sections work without explicit membership |

## Recovery

### Your OIDC subject doesn't match the admin record

You signed in but you don't see Cluster Administration. The user record exists, but its `oidc_subject` is something other than your real subject.

Re-apply the `apps` stack with the correct value:

```sh
cd bootstrap
export ADMIN_OIDC_SUBJECT=<your real subject>
terraform -chdir=stacks/apps apply -var "admin_oidc_subject=$ADMIN_OIDC_SUBJECT"
```

This recreates the admin user with the new subject. Existing data is unaffected.

### You need a second cluster admin

Sign in as the existing cluster admin, open Console → **Cluster Administration → Users**, find the second user, and toggle **Cluster Admin** on. Or do it via Terraform / the Gateway API.

This is recommended — single-admin setups are fragile.

### You lost the admin sign-in

If you cannot sign in as the admin and don't have a second admin:

- **Bootstrap install with the mock IdP**: sign in as `admin@agyn.io` via the mock IdP. The credentials are well-known.
- **Real OIDC**: re-run the `apps` stack with `ADMIN_OIDC_SUBJECT` set to a subject you can sign in as.
- **Last resort**: write the OpenFGA tuple by hand. Find the user's identity_id (Postgres `users` table) and create the tuple `identity:<id>, admin, cluster:global` against the OpenFGA store. Restart Authorization. You then have cluster admin on next page load.

The bootstrap `cluster_admin_api_token` Terraform output is also a recovery path — you can use it as a bearer token against Gateway to grant cluster admin to another identity programmatically. Retrieve it with:

```sh
terraform -chdir=bootstrap/stacks/platform output -raw cluster_admin_api_token
```

## Hardening

Once you are signed in as yourself:

1. Replace the mock OIDC with a real IdP. Re-apply the `platform` and `apps` stacks with real values.
2. Create at least one backup cluster admin.
3. Rotate the static `cluster_admin_api_token` if you do not need it for operations (delete it from state and re-run, or revoke through the Users service).
4. Restrict who in your IdP can be assigned the admin OIDC subject.
